# Plan: Fix Email Sending Issues

## Root Cause Analysis

### Issue 1: Forgotten Password Page is a Stub / Mock (ROOT CAUSE)
`app/forgot-password/page.tsx` is demo/placeholder code — it never calls any backend. Line 46 uses `setTimeout(resolve, 1500)` and logs to `console.log` instead of actually sending an email:
```
// In production, this would call an API to send a reset email
// For demo, we simulate a successful request
await new Promise((resolve) => setTimeout(resolve, 1500))
console.log("Reset password for:", data.email)
setSuccess(true)
```
There is **no** `app/api/auth/forgot-password/route.ts`, **no** `app/api/auth/reset-password/route.ts`, no `app/reset-password/page.tsx`, and the `User` model has no `resetPasswordToken` / `resetPasswordExpiry` fields. The entire password reset flow is fake.

### Issue 2: Email sending in signup route is unguarded
`lib/email.ts` has no `try/catch` blocks. If the nodemailer SMTP transport throws (auth failure, connectivity error, etc.), the unhandled promise rejection propagates to the calling `/api/auth/signup` route handler, killing the HTTP 201 response. The user's account is silently created in MongoDB (`User.create()` runs line 25 before `sendVerificationEmail` line 38 in signup route), so the user account persists, but the response becomes HTTP 500 with no verification email ever sent and no log of why.

### Issue 3: Brevo SMTP credentials may have issues in production
`BREVO_USER` / `BREVO_KEY` are sourced from `process.env`. In Vercel/netlify serverless function deployments the env vars must be set in the dashboard and must not be prefixed `NEXT_PUBLIC_`. If missing, nodemailer initially connects (line 3 via `createTransport`) but throws on the actual `.sendMail()` call — which currently goes uncaught.

## Files to Change

### 1. `lib/email.ts` — Add error wrapping + defined logging
Add `try/catch` to both `sendVerificationEmail` and `sendNotificationEmail`. Log a descriptive error to `console.error` so SMTP failures are diagnosable without crashing the HTTP flow.

### 2. `app/api/auth/signup/route.ts` — Guard email send against failure
Wrap `sendVerificationEmail` in `try/catch` so the account is still created + returned to the caller even if the email silently fails.

### 3. `models/User.ts` — Add password reset token fields
Add `resetPasswordToken?: string` and `resetPasswordExpiry?: Date` to schema and TypeScript interface.

### 4. `app/api/auth/forgot-password/route.ts` — NEW
New `POST` route: accept `email`, find user, generate/reset token, save expiry (30 min window), send email, always return generic success to prevent user enumeration.

### 5. `app/api/auth/reset-password/route.ts` — NEW
New `POST` route: accept `token` + `newPassword`, validate token freshness, update user password, clear token.

### 6. `app/forgot-password/page.tsx` — Replace stub with real API call
Remove the mock `setTimeout` code and replace with `fetch("/api/auth/forgot-password")`.

### 7. `app/reset-password/page.tsx` — NEW
New page for the password reset form (URL: `?token=<token>`).

### 8. `app/reset-password/sent/page.tsx` — NEW (optional)
Confirmation page after forgot-password email sent — distinct from the stub present on forgot-password page itself.
